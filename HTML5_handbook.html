01

The Boilerplate

Every HTML document starts here. Memorize what it does, not the characters — you'll never type it from scratch by Week 3.

html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="description" content="A short, honest summary of the page for search engines">
  <title>Page Title</title>
  <link rel="icon" href="favicon.ico">
  <link rel="stylesheet" href="styles.css">
</head>
<body>

  <!-- your content -->

  <script src="app.js" defer></script>
</body>
</html>
<!DOCTYPE html> — tells the browser "render this in standards mode." Always the first line, no exceptions.
<html lang="en"> — the lang attribute isn't decoration; screen readers use it to pick pronunciation, and browsers use it for translation prompts.
<script defer> — loads the script in the background and runs it only after the HTML is fully parsed, in order. This is why <script> used to go right before </body> — with defer, you don't even need to; it can live in <head>.
02

Document Metadata

Everything inside <head> — nothing here renders on the page itself, but all of it matters.

<head>

Container for metadata — nothing here is visible on the page itself.

<title>

The browser tab text and default bookmark/search-result title — every page needs exactly one.

<meta charset>

Character encoding — always include this, first thing in <head>.

<meta viewport>

Makes the page render correctly on mobile instead of zoomed-out desktop-style.

<meta description>

The text Google shows under your link in search results.

<link rel="stylesheet">

Attaches an external CSS file.

<link rel="icon">

The favicon.

<style>

Inline CSS block — fine for small demos, avoid in real projects. Use external files instead.

<script>

Embeds or links JavaScript. Use defer (runs after parsing, in order) or async (runs as soon as loaded, order not guaranteed) — plain <script> with no attribute blocks HTML parsing until it finishes, which is the thing you're trying to avoid.

<base href>

Sets a default base URL for every relative link on the page — rare, but explains weird link bugs when it's present and you didn't know it.

<noscript>

Fallback content shown only if JavaScript is disabled. (You're reading its output right now, at the bottom of this page, if JS is off.)

03

Sectioning & Semantic Structure

This is the part most beginners get sloppy on — and the part that actually matters for accessibility and SEO. Rule of thumb: reach for a semantic tag first; only fall back to <div> when nothing else fits.

<body>

Everything visible on the page.

<header>

Introductory content — a logo, nav, page heading. Can appear once per page or once per <article>/<section>.

<nav>

A block of primary navigation links — not every group of links needs one.

<main>

The one central, unique content of the page — exactly one per page.

<article>

Self-contained content that would make sense distributed on its own — a blog post, a product card, a comment.

<section>

A thematic grouping of content, usually with its own heading — used when <article> is too specific but <div> is too meaningless.

<aside>

Content tangentially related to the main content — a sidebar, a pull quote, related links.

<footer>

Closing content — copyright, contact info, site links.

<h1>–<h6>

Headings, in strict order of importance. Don't skip levels (h2 straight to h4) — screen reader users navigate by heading structure.

<hgroup>

Groups a heading with its subheading so assistive tech treats them as one unit.

<address>

Contact information for the page/article author — has real semantic meaning, not just a styling hook.

<div>

A generic block container with no semantic meaning — the fallback when nothing above fits, not the default.

<search>

A newer (2024+) landmark for wrapping search/filter functionality — genuinely worth using over a bare <div class="search">.

The skeleton in practice
html
<body>
  <header>
    <nav><!-- site links --></nav>
  </header>

  <main>
    <article>
      <header><h1>Post Title</h1></header>
      <p>Content...</p>
    </article>
    <aside>Related posts</aside>
  </main>

  <footer>&copy; 2026</footer>
</body>
Structure, mapped
header → nav
main
article → header + h1 + p
aside → related posts
footer → © 2026

05

Inline Text Semantics

These live inside a line of text. The distinction beginners miss: several of these look identical to CSS styling but carry real meaning — screen readers announce them differently, and search engines weight them differently.

<a>

A hyperlink.

Big enough to deserve its own section — see §06 below.

<strong>

Strong importance.

Renders bold — but the meaning, not the bold, is the point.

<em>

Stress emphasis (changes meaning when spoken aloud).

Renders italic, same distinction as <strong>.

<b>

Draws attention without extra importance — product names, keywords.

Use <strong> if it's actually important.

<i>

An alternate voice/mood — technical terms, taxonomic names, foreign phrases.

Use <em> if it's actually emphasis.

<u>

An unarticulated annotation — e.g. flagging a misspelling.

Rarely what you want: underline visually implies "link," so don't use this just to underline text for style.

<small>

Side comments, fine print, disclaimers.

Not just "smaller text" — has legal/print connotation.

<s>

Content that is no longer accurate or relevant — a struck-through price.

Different from the old <strike> tag, which is deprecated.

<mark>

Highlighted text relevant to the current context — like a search-term highlight.

<cite>

The title of a creative work — a book, article, song.

<q>

A short inline quotation — browsers add quote marks automatically.

Use <blockquote> for longer, block-level quotes.

<dfn>

Marks the defining instance of a term.

<abbr>

An abbreviation — use the title attribute for the full form.

<abbr title="Cascading Style Sheets">CSS</abbr>

<time>

A machine-readable date/time via the datetime attribute.

<time datetime="2026-07-02">July 2, 2026</time>

<data>

A machine-readable value via the value attribute.

<data value="128">One-Twenty-Eight</data>

<code>

A snippet of computer code.

<var>

A variable name — math or programming context.

<samp>

Sample output from a program.

<kbd>

Keyboard input the user should type.

Press <kbd>Ctrl</kbd>+<kbd>S</kbd>

<sub> / <sup>

Subscript / superscript.

H<sub>2</sub>O

<ruby> <rt> <rp>

Pronunciation annotations for East Asian typography.

Niche, but real.

<bdi> / <bdo>

Isolate or override text direction, for mixed LTR/RTL content.

Niche, but real.

<span>

A generic inline container with no semantic meaning — the inline equivalent of <div>.

<br>

A line break within content — a poem line, a mailing address.

Never use it to fake vertical spacing — that's CSS's job.

<wbr>

A suggested word-break point in a long unbreakable string — URLs, long words.

06

Links — <a> Deep Dive

One tag, five very different behaviors depending on the href.

html
<a href="/about">Relative link — same site</a>
<a href="https://example.com">Absolute link — external site</a>
<a href="#section-2">Anchor link — jumps to id="section-2"</a>
<a href="mailto:hello@example.com">Opens the user's email client</a>
<a href="tel:+911234567890">Opens the phone dialer (mobile)</a>
<a href="https://example.com" target="_blank" rel="noopener noreferrer">
  Opens in a new tab, safely
</a>
Live — these actually work
↳ Anchor link — jumps to the Tables section ✉ mailto: — opens your email client ☎ tel: — opens the phone dialer ↗ External, opens safely in a new tab
target="_blank" always pair with rel="noopener noreferrer" — without it, the new tab gets partial access back to your original page's window object. That's a real, documented security issue, not just paranoia.
rel="nofollow" tells search engines not to pass ranking credit through the link — used for user-submitted or sponsored links.
07

Images & Responsive Media

One image tag, plus the machinery for serving the right version to the right screen.

html
<img src="photo.jpg" alt="A golden retriever running on a beach" width="800" height="600" loading="lazy">
alt is not optional. Meaningful image → describe it. Purely decorative image → alt="" (empty, not missing) so screen readers skip it silently.
width/height reserve the space before the image loads, preventing the page from jumping around as images pop in — a real Core Web Vitals metric.
loading="lazy" defers off-screen images until the user scrolls near them.
Responsive / art-directed images

Serve a different image per screen size or format:

html
<picture>
  <source media="(min-width: 800px)"
          srcset="hero-desktop.webp" type="image/webp">
  <source media="(min-width: 800px)"
          srcset="hero-desktop.jpg">
  <img src="hero-mobile.jpg" alt="Description">
</picture>
How the browser decides
≥ 800px viewport
→ tries hero-desktop.webp, falls back to .jpg
< 800px viewport
→ falls through to the plain <img>, hero-mobile.jpg
<map> / <area>

Clickable regions inside a single image — rare today, but you'll see it in older codebases.

Audio, Video & Embedded Content

Media playback and ways to embed content that isn't yours.

html
<video controls width="640" poster="thumbnail.jpg">
  <source src="movie.webm" type="video/webm">
  <source src="movie.mp4" type="video/mp4">
  <track kind="subtitles" src="subs-en.vtt" srclang="en" label="English">
  Your browser doesn't support video.
</video>

<audio controls>
  <source src="song.mp3" type="audio/mpeg">
</audio>
Multiple <source> children let the browser pick the first format it supports.
<track> adds captions/subtitles — don't skip this, it's an accessibility requirement, not a nice-to-have.
Autoplay video must also be muted or most browsers will silently block it anyway.
<iframe>

Embeds another page/document — always give it a title attribute for accessibility.

<embed>

Embeds external content via a plugin (PDFs, etc.) — increasingly rare.

<object>

Similar to <embed>, more general-purpose.

<canvas>

A blank drawing surface controlled entirely via JavaScript.

<svg>

Scalable vector graphics — can be written directly inline in HTML.

Tables — Full Structure

Tables are for tabular data only — never for page layout. That was a widespread mistake in the late-1990s/2000s web, and it's exactly the kind of outdated pattern this whole guide steers you away from.

html
<table>
  <caption>Q1 2026 Revenue by Region</caption>
  <colgroup>
    <col style="background-color: #f5f5f5">
    <col span="2">
  </colgroup>
  <thead>
    <tr>
      <th scope="col">Region</th>
      <th scope="col">Revenue</th>
      <th scope="col">Growth</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">North</th>
      <td>₹12L</td>
      <td>+8%</td>
    </tr>
  </tbody>
  <tfoot>
    <tr>
      <th scope="row">Total</th>
      <td>₹12L</td>
      <td>+8%</td>
    </tr>
  </tfoot>
</table>

scope="col"/scope="row" on <th> tells screen readers which header applies to which cell — small addition, real accessibility impact.
<caption> is the table's accessible title — better than a floating <p> above it.

Forms — The Big One

The most attribute-heavy corner of HTML, and the one that talks directly to your backend.

html
<form action="/submit" method="post" novalidate>

  <fieldset>
    <legend>Account Details</legend>

    <label for="email">Email</label>
    <input type="email" id="email" name="email" required autocomplete="email">

    <label for="password">Password</label>
    <input type="password" id="password" name="password" minlength="8" required>
  </fieldset>

  <fieldset>
    <legend>Plan</legend>
    <label><input type="radio" name="plan" value="free"> Free</label>
    <label><input type="radio" name="plan" value="pro"> Pro</label>
  </fieldset>

  <label for="bio">Bio</label>
  <textarea id="bio" name="bio" rows="4"></textarea>

  <button type="submit">Create Account</button>
  <button type="reset">Clear</button>
</form>
Non-negotiable rule: every <input> needs an associated <label> — either via matching for/id, or by wrapping the input. This isn't optional politeness; it's what lets a screen reader announce the field's purpose, and it's also what makes clicking the label text focus the input.

Plan
<input> types — one element, wildly different behavior

Common attributes: name (required — the key the value submits under), value, placeholder (a hint, never a replacement for <label>), required, disabled, readonly, min/max/step, pattern (regex), maxlength/minlength, autocomplete, autofocus, multiple.

The <button> gotcha: inside a <form>, a <button> with no type attribute defaults to type="submit". Classic bug: a "cancel" or "toggle" button silently submits the form. Always set type="button" explicitly unless you actually mean to submit.
<select> <option> <optgroup>

Dropdown menu, optionally grouped.

<datalist>

A list of autocomplete suggestions, linked to an input via the list attribute.

<output>

Displays the result of a calculation — for links it to the inputs involved.

<progress>

A task's completion progress (value/max) — how much is done.

<meter>

A measurement within a known range — a gauge, not a progress bar. Commonly confused with <progress>: the distinction is completion vs. measurement.

<fieldset> <legend>

Groups related fields with an accessible group label — essential for radio/checkbox groups.

Interactive Elements (No JavaScript Required)

Genuine interactivity, shipped natively by the browser.

html
<details>
  <summary>Click to expand</summary>
  <p>Hidden content, revealed natively — no JS
     accordion library needed.</p>
</details>

<dialog id="myModal">
  <p>A native modal dialog.</p>
  <button onclick="myModal.close()">Close</button>
</dialog>
<button onclick="myModal.showModal()">Open Modal</button>
Live
Click to expand

Hidden content, revealed natively — no JS accordion library needed.

<dialog> handles focus-trapping and comes with a ::backdrop you can style — genuinely replaces a lot of what people used to reach for a JS modal library to do.

Worth knowing about — genuinely current, not in most older tutorials

The popover global attribute reached full cross-browser support in April 2025. It turns any element into a native floating overlay — dropdown, toast, tooltip — with automatic top-layer stacking, click-outside dismissal, and Esc-to-close, all without JavaScript or a library like Popper/Floating UI:

html
<button popovertarget="menu">Open menu</button>
<div id="menu" popover>
  <a href="/profile">Profile</a>
  <a href="/settings">Settings</a>
</div>

Global Attributes

Work on almost any tag, regardless of what it is.

id

A unique identifier — used for CSS, JS, label for=, anchor links.

class

One or more style/JS hooks, space-separated.

style

Inline CSS — fine for quick tests, avoid in real projects.

title

A tooltip shown on hover.

data-*

Custom data attributes for your own JS to read.

data-user-id="42"

tabindex

Controls keyboard focus order — 0 inserts into natural tab order, -1 makes it focusable only via JS, positive numbers are almost always a mistake.

hidden

Hides the element entirely (boolean attribute).

contenteditable

Makes the element's content directly editable by the user.

draggable

Enables native drag-and-drop.

lang

Language of the element's content — affects screen reader pronunciation and translation tools.

dir

Text direction: ltr, rtl, or auto.

spellcheck

Toggles the browser's spellcheck on editable content.

role

An ARIA role — your gateway into accessibility semantics beyond what native tags cover.

13

Accessibility Essentials

A handful of rules that cost nothing and matter enormously. Click each one off as you internalize it.

Every meaningful <img> gets a real alt. Purely decorative images get alt="" — never just omit the attribute.

Every <input> gets a real <label>. A placeholder is not a label; it disappears the moment someone starts typing.

Headings go in strict order. Don't skip from h2 to h4 just because it looks smaller — screen reader users navigate a page by jumping between headings, and skipped levels break that mental map.

Prefer native semantics over ARIA. <button> is already keyboard-accessible and announced correctly; a <div role="button"> requires you to manually replicate all of that. The community's own guideline: "no ARIA is better than bad ARIA."

aria-label gives an element an accessible name with no visible text (an icon-only close button); aria-labelledby points to another element's id to reuse as the label.

Every <iframe> gets a title. (Check §08 above — the live iframe demo has one.)

Quick Semantic Decision Cheat Sheet

When you're not sure which tag to reach for, walk down this list:

<nav>

Is it the page's single most important navigation block?

<article>

Is it standalone content that would make sense syndicated on its own — a post, a card, a comment?

<section>

Is it a thematic chunk with its own heading, but not standalone?

<aside>

Is it related but tangential — sidebar, pull-quote?

<header>

Is it intro content at the top of the page or an article?

<footer>

Is it closing content — copyright, links, contact?

<div> / <span>

None of the above, and you just need a box to hook CSS/JS onto? <div> — or <span> if it's inline.

If you can name why content belongs together in a sentence — "this is the navigation," "this is a self-contained article" — there's almost always a semantic tag for it. <div> is the answer only when there genuinely isn't one.
