The Mental Model: Cascade, Specificity, Inheritance

Read this first — it explains why your CSS does what it does, which matters more than memorizing any single property.

The cascade — who wins when several rules apply
1. Importance

!important beats everything — which is exactly why overusing it causes chaos.

2. Specificity

Inline style (1000) > #id (100) > .class, [attr], :pseudo-class (10) > element, ::pseudo-element (1). The universal selector * has zero specificity.

3. Source order

Later rules win ties.

The Box Model & display

Every element is a box: content → padding → border → margin, from the inside out.

css
* {
  box-sizing: border-box; /* width/height include padding + border.
                              Set this globally — it's what you want,
                              essentially always. */
}
Without border-box (the default, content-box), width: 200px plus padding: 20px renders as a 240px-wide box — a constant source of layout bugs for beginners. Try it below.

Margin collapsing — the classic gotcha

Two vertical margins between adjacent block siblings collapse into a single margin (the larger of the two), not the sum. Doesn't happen with flex/grid children, or horizontally.

04

Units, Values & Functions

What your numbers are actually measured against.

px

Absolute pixel.

%

Parent's size.

em

The element's own font-size — compounds when nested, a common surprise.

rem

The root (<html>) font-size — usually what you actually want for consistency.

vw / vh

1% of viewport width/height.

dvh / svh / lvh

Dynamic/small/large viewport height — solves the classic mobile bug where 100vh gets covered by the browser's address bar. Use dvh for full-height mobile layouts now.

ch

Width of the 0 character — useful for readable line-length limits.

Functions worth knowing cold
css
width: calc(100% - 2rem);        /* mix units in one expression */
width: min(90%, 600px);          /* whichever is smaller */
width: max(300px, 50%);          /* whichever is larger */
font-size: clamp(1rem, 2vw + 1rem, 2.5rem); /* fluid: min, preferred, max */
clamp() is the single highest-value function for responsive typography and spacing — it replaces a whole tier of @media breakpoints for font-size scaling. Drag below to see it work.

Typography

Text is the most inherited, most read, most fussed-over part of any page.

Flexbox — One-Dimensional Layout

Arranges items along a single axis — row or column. This is what powers a navbar or a row of cards.

css
.container {
  display: flex;
  flex-direction: row;            /* row | row-reverse | column | column-reverse */
  justify-content: space-between; /* main-axis alignment */
  align-items: center;            /* cross-axis alignment */
  flex-wrap: wrap;                /* nowrap | wrap | wrap-reverse */
  gap: 1rem;                      /* prefer this over margin hacks */
}
.item {
  flex-grow: 1;         /* how eagerly this item grows */
  flex-shrink: 1;        /* how eagerly it shrinks */
  flex-basis: 200px;     /* starting size before grow/shrink */
  flex: 1 1 200px;       /* shorthand — 90% of the time, this */
  align-self: flex-end;  /* override align-items for one item */
  order: 2;               /* visually reorder, no HTML change */
}

CSS Grid — Two-Dimensional Layout

Arranges items in rows and columns simultaneously. This is what powers a full dashboard layout.

css
.container {
  display: grid;
  grid-template-columns: repeat(3, 1fr);                      /* fr = fraction of remaining space */
  grid-template-columns: repeat(auto-fit, minmax(200px, 1fr)); /* responsive, zero media queries */
  grid-template-rows: auto 1fr auto;
  gap: 1.5rem;
}

Named areas — the most readable way to build a real layout
css
.dashboard {
  display: grid;
  grid-template-columns: 240px 1fr;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "sidebar header"
    "sidebar main"
    "sidebar footer";
}
.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }
Manual placement
css
.item {
  grid-column: 1 / 3;   /* start at line 1, end at line 3 — spans 2 columns */
  grid-row: span 2;      /* spans 2 rows from wherever it lands */
}
Subgrid (Baseline since 2023) lets a nested grid's items align to the parent grid's tracks — a row of cards with uneven internal content can still line up their headings/buttons across the row: grid-template-rows: subgrid;
09

Positioning

Takes an element out of (or keeps it in) normal flow.

css
position: static;    /* default — normal document flow */
position: relative;  /* stays in flow, offsets from where it WOULD be */
position: absolute;  /* removed from flow, positioned relative to nearest
                         non-static ancestor */
position: fixed;      /* relative to the viewport, stays put on scroll */
position: sticky;     /* relative until a scroll threshold, then fixed */
top/right/bottom/left only do anything on non-static elements. If z-index "isn't working," the real problem is almost always a stacking context trap: a parent with opacity, transform, or filter silently creates a new stacking context.

Responsive Design: Media & Container Queries

Two different questions: "how big is the screen?" vs. "how big is the space I've been given?"

css
/* Mobile-first: base styles are mobile, media queries add complexity up */
.grid { grid-template-columns: 1fr; }

@media (min-width: 768px) {
  .grid { grid-template-columns: repeat(2, 1fr); }
}
@media (min-width: 1024px) {
  .grid { grid-template-columns: repeat(3, 1fr); }
}
css
/* Container queries (Baseline since 2023) respond to a component's
   OWN container width, not the screen */
.card-wrapper { container-type: inline-size; }

@container (min-width: 400px) {
  .card { display: grid; grid-template-columns: 120px 1fr; }
}
This is the fix for the classic "this card looks great in the main grid but breaks when a designer drops it into a narrow sidebar" problem. Rule of thumb: media queries for page-level layout, container queries for reusable components.

Motion, done cheaply — animate transform, not top/left/width.

Custom Properties (CSS Variables)

Real runtime values — unlike Sass variables, you can read and change them with JavaScript, and override them per-component by redeclaring in a narrower scope. The actual mechanism behind most theme-switchers.

css
:root {
  --color-primary: #6366f1;
  --spacing-md: 1rem;
  --radius: 12px;
}
.button {
  background: var(--color-primary);
  padding: var(--spacing-md);
  border-radius: var(--radius, 8px); /* fallback if undefined */
}

Modern Cascade & Architecture Tools

How current codebases stay organized without a preprocessor.

css
/* Native CSS Nesting — Baseline, all major engines. No build step. */
.card {
  padding: 1rem;
  & h2 { font-size: 1.5rem; }
  &:hover { box-shadow: 0 4px 8px rgba(0,0,0,0.1); }
  &__header { border-bottom: 1px solid #eee; }
}
Keep nesting to 2 levels or so — it's readable formatting, not a license to rebuild deep specificity chains.
Cascade Layers (@layer)

Explicitly control which group of rules wins, independent of specificity or source order. This is how modern codebases end specificity wars instead of reaching for !important.

css
@layer reset, base, components, utilities;

@layer base {
  h1 { font-size: 2rem; }
}
@layer components {
  .card h1 { font-size: 1.5rem; } /* wins over @layer base even with
                                      lower specificity — layer order
                                      decides first */
}
Logical properties — live

Write layout that adapts automatically to right-to-left languages, instead of hardcoding left/right.

css
margin-inline: 1rem;   /* instead of margin-left + margin-right */
padding-block: 2rem;    /* instead of padding-top + padding-bottom */

Common Gotchas & Debugging

Symptom on the left, real cause on the right.

"My margin isn't the size I set"

Margin collapsing (§03), or a parent with padding/border you forgot about.

"z-index isn't working"

Check for a stacking-context trap: a parent with opacity, transform, or filter silently creates a new stacking context that traps your z-index inside it.

"My flex items won't shrink below their content size"

Set min-width: 0 on the flex item — flex items default to min-width: auto, which can force overflow.

em compounding

Nested elements each multiply em by their parent's font-size. Use rem unless you specifically want that compounding behavior.

!important everywhere

Almost always a specificity or cascade-layer problem being papered over. Fix the actual specificity instead of escalating.

18

Quick Decision Cheat Sheets

Three questions you'll ask yourself constantly.

Flexbox or Grid?

One dimension (a row of nav links, a stack of cards) → Flexbox. Two dimensions at once (a full page layout, a dashboard) → Grid. They combine constantly: Grid for the page skeleton, Flexbox inside individual components.

rem, em, or px?

Font sizes, spacing, layout dimensions → rem. Something that scales with its own element's font-size (icon next to text, padding inside a button) → em. Borders, and only borders → px is genuinely fine.

Media or container query?

Whole-page layout decisions → media query. A reusable component that needs to adapt to wherever it's dropped → container query.

Everything above is what Tailwind's utility classes (flex, grid-cols-3, p-4, rounded-xl) compile down to under the hood — so it'll feel like a shorthand you already understand, not a new language.
